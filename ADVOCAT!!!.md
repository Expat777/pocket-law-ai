# 🏛️ Карманный юрист — Telegram-бот по законодательству РФ

AI-агент для консультаций по законодательству РФ с проверяемыми ссылками на первоисточники. Пользователь задаёт вопрос или присылает документ (например, договор) — бот отвечает со ссылками на конкретные статьи и пункты нормативных актов, помечая дату актуальности.

> **Дисклеймер.** Проект носит справочный характер и **не заменяет консультацию квалифицированного юриста**. Ответы формируются на основе базы нормативных актов и могут содержать неточности. Для юридически значимых решений обращайтесь к профильному специалисту.

## 🎯 Идея и статус

Локальный по духу, но серверный по реализации помощник по праву РФ. Первоначально задумывался как офлайн desktop-приложение (Tauri), но для соблюдения срока в 2 недели команда выбрала **Telegram-бот** как основной интерфейс: нулевой онбординг для пользователя, простая загрузка документов, лёгкая демонстрация. Desktop-версия обозначена как следующий шаг для сценариев, где критична полная приватность.

**Статус:** MVP / учебный проект. Срок — 2 недели, команда — 4 человека.

**Scope MVP:** 2–3 кодекса РФ (а не «всё законодательство»), RAG-поиск с проверкой цитат, обработка PDF-документов пользователя, полуавтоматическое обновление базы.

## ✨ Ключевые возможности

- **Вопрос-ответ по праву РФ** с обязательными ссылками на статью/пункт и датой актуальности нормы.
- **Проверка цитат (anti-hallucination).** Отдельный узел графа сверяет каждую цитату с реально найденными фрагментами базы — агент не выдумывает нормы.
- **Уточняющие вопросы.** При недостатке контекста бот переспрашивает пользователя, а не додумывает ответ.
- **Загрузка документов.** Пользователь присылает PDF/фото (например, договор) — бот извлекает текст (OCR при необходимости) и анализирует его в контексте законодательства.
- **Диалоговая память.** Состояние разговора хранится по `telegram_user_id` — бот помнит контекст при уточнениях.
- **Обновление базы.** Скрипт-DAG подгружает свежие редакции НПА и формирует diff-отчёт об изменениях.
- **Проактивные уведомления (roadmap).** Пуш в Telegram, если норма, на которую пользователь ранее ссылался, изменилась.

```
        ┌───────────────────────────────────────────────┐
        │ ИНФРАСТРУКТУРА (Selectel VM)                     │

```

Полная редактируемая схема — в файлах `docs/architecture_final.excalidraw` / `.svg` / `.png`.

**Ключевые архитектурные решения:**

| Решение | Обоснование |
|---|---|
| Telegram-бот вместо desktop | Уложиться в 2 недели, нулевой онбординг, простая загрузка файлов |
| Bot + LangGraph в одном docker-сервисе | Меньше движущихся частей при деплое на старте |
| Внешний LLM API (не локальная модель) | Качество на юридических формулировках; локальная модель — как fallback в будущем |
| MCP-инструменты с read-only на Qdrant | Изоляция: писать в базу может только Data Pipeline, инструменты поиска — только читать |
| `insufficient_context` → вопрос пользователю | Убирает риск бесконечного внутреннего цикла и галлюцинаций при неполном контексте |

## 🧠 Как работает агент (LangGraph)

Граф состояний обрабатывает каждый запрос по узлам:

1. **`intent_classifier`** — нормализует вопрос, определяет отрасль права. При неоднозначности может запросить уточнение.
2. **`retrieve`** — гибридный поиск (векторный + BM25) по Qdrant через инструмент `search_law`.
3. **`verify_citation`** — сверяет кандидатов-цитаты с найденными фрагментами: проверяет и **существование** нормы, и её **актуальность** (`status`, `effective_date`). Защита от галлюцинаций.
4. **`insufficient_context?`** — условный узел. Если данных недостаточно — граф направляет уточняющий вопрос пользователю через Response Formatter, а не выдумывает ответ.
5. **`compose_answer`** — обращается к внешнему LLM API, собирает финальный ответ: текст + ссылка на статью + дисклеймер.

Обработка загруженного документа: `File Handler → parse_pdf (MCP) → извлечённый текст → compose_answer`, параллельно текст пользователя эмбеддится и пишется в коллекцию `user_documents` (изолированную по `user_id`, с TTL).

## 🔄 Data Pipeline — обновление базы

Реализовано как **обычный Python-скрипт с логикой Airflow** (идемпотентные функции-шаги), без поднятия самого Airflow на старте — чтобы не блокировать разработку инфраструктурой. Обёртка в полноценный Airflow DAG — опциональный шаг, если останется время.

```
fetch          # выгрузка НПА с официальных порталов (pravo.gov.ru и др.)
  → parse      # TXT/HTML + PDF/OCR, сохранение структуры статья/пункт
  → chunk      # разбиение по смысловым границам (не механически)
  → embed      # эмбеддинги фрагментов
  → upsert     # запись в Qdrant (law_articles)
  → diff       # отчёт об изменениях; при изменении нормы → событие для пуша
```

Запуск на старте — по `cron` или ручным триггером. Поля `status` и `effective_date` в метаданных позволяют отличать действующие нормы от утративших силу.

## 🔐 Безопасность и приватность

> ⚠️ Поскольку бот работает через сервер и принимает документы пользователей, проект де-факто обрабатывает **персональные данные** (152-ФЗ). Ниже — минимальные меры; для продакшена потребуется полноценное соответствие (уведомление РКН, политика обработки ПДн, явное согласие).

- **Согласие на обработку ПДн** при `/start` — обязательное условие использования.
- **Дисклеймер в каждом ответе** с юридической рекомендацией (а не только при старте — пользователи не читают приветствие).
- **Минимизация хранения.** Загруженные PDF удаляются после обработки сессии; содержимое файлов не логируется.
- **Валидация файлов** по magic bytes, а не только по расширению — защита от подмены типа.
- **Изоляция MCP-инструментов.** Whitelist прав; инструменты поиска имеют read-only доступ к Qdrant.
- **Rate-limit** на пользователя (например, 20 запросов/день) — включён с первого дня деплоя, защита от злоупотреблений и перерасхода бюджета на LLM API.
- **Шифрование** данных пользователей на сервере.
- **Бэкапы** (`pg_dump` + снапшот Qdrant) — регулярно, т.к. единая VM является точкой отказа.

## 🚀 Установка и запуск

### Требования

- Docker и Docker Compose
- Python 3.11+ (для локальной разработки вне контейнеров)
- Токен Telegram-бота (от [@BotFather](https://t.me/BotFather))
- Ключ внешнего LLM API

### Быстрый старт (Docker)

```bash
# 1. Клонировать репозиторий
git clone <repo-url> pocket-lawyer && cd pocket-lawyer

# 2. Настроить окружение
cp .env.example .env
# отредактировать .env — вписать токены и ключи

# 3. Поднять сервисы (bot + Qdrant + Postgres)
docker-compose up -d

# 4. Инициализировать базу законодательства
docker-compose exec bot python -m data_pipeline.pipeline --init

# 5. Проверить логи
docker-compose logs -f bot
```

### Режимы работы бота

| Режим | Когда использовать | Требования |
|---|---|---|
| **Polling** | Разработка, локальные тесты | Ничего дополнительного |
| **Webhook** | Демо, продакшен | Домен + HTTPS (Let's Encrypt) |

Для разработки достаточно `polling` (запускается за минуты, все в команде тестируют локально). К демо — переключение на `webhook` на Selectel VM.

## 👥 Роли команды

| Роль | Зона ответственности | Ключевые задачи |
|---|---|---|
| **Agent-инженер** | LangGraph-граф и MCP-инструменты | Граф состояний, промпт-инжиниринг против галлюцинаций, MCP-серверы, тесты качества |
| **Bot/Frontend-инженер** | Telegram-интерфейс | aiogram, FSM диалога, приём/валидация файлов, форматирование, онбординг, дисклеймеры |
| **Data-инженер** | База законодательства | Источники, парсинг TXT/HTML/PDF+OCR, чанкинг, эмбеддинги, скрипт-DAG обновления |
| **DevOps/Security/QA** | Инфраструктура и качество | Деплой на Selectel, webhook/HTTPS, изоляция MCP, rate-limit, шифрование, ручная сверка юр. кейсов, бэкапы |

**Точки синхронизации:** контракт «запрос → ответ» между Bot и Agent, формат метаданных чанков между Agent и Data, размещение БД между Data и DevOps. Рекомендуется зафиксировать JSON-контракты между слоями в первый день.

## 📅 План разработки (2 недели)

Занятость 8 ч/будни → **10 рабочих дней**, ~320 человеко-часов.

| Дни | Фокус |
|---|---|
| **1–2** | Источники, скелеты (бот, граф, docker-compose на Selectel), контракты между слоями, согласие на ПДн |
| **3–5** | Параллельно: парсинг+эмбеддинги первого кодекса; граф retrieve+verify; диалог + приём файлов; webhook/HTTPS + rate-limit |
| **6–7** | Полный граф, PDF/OCR-пайплайн, MCP-инструменты, first end-to-end, второй кодекс, дисклеймеры в ответах |
| **8** | Ручная юридическая проверка (15–20 кейсов), доработка промптов, edge-cases |
| **9** | Security-обзор, стабилизация, опциональная обёртка DAG в Airflow |
| **10** | Сквозное тестирование, финальный бэкап, подготовка демо |

**Контрольная точка недели 1:** рабочий сценарий «вопрос → ответ с цитатой» хотя бы на одном кодексе.

## ⚠️ Известные ограничения

- **Юридическая точность.** В команде нет профильного юриста — точность ответов проверяется вручную по открытым источникам (15–20 кейсов). Не полагаться на бота в юридически значимых ситуациях.
- **Покрытие базы.** MVP охватывает 2–3 кодекса, а не всё законодательство РФ.
- **OCR кириллицы.** Tesseract на плохих сканах даёт заметно худшее качество, чем на английском — для сложных сканов предусмотрен fallback на Yandex Vision.
- **Лимит файлов Telegram.** Bot API принимает файлы до 20 МБ — многостраничные сканы могут не пройти.
- **Единая точка отказа.** Всё на одной VM — критична регулярность бэкапов.
- **Соответствие 152-ФЗ.** Реализованы лишь минимальные меры; полноценное соответствие — вне scope MVP.
