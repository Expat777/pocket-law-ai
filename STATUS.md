### 2026-07-08 — план мульти-акт (Tier 1: ЗоЗПП + СК РФ) · решение по хранению
**Решение: ОДНА коллекция `law_articles`, роутинг по полю `act` — НЕ режем на коллекции-на-закон.** Почему так эффективнее агенту: при отдельных коллекциях агент обязан выбрать базу ДО поиска, ошибка отрасли = жёсткий промах; при одной — «не уверен» = ищем без фильтра по всем, нужный закон всплывает сам (косинусы bge-m3 сравнимы между актами, одна модель + normalize). Мягкая деградация вместо промаха, стыковые вопросы (ТК+СК) бесплатны, инфра минимальна. Единственный плюс раздельных коллекций (своя модель/чанкинг на закон) нам не нужен — всё русский юртекст.

Прод-схема:
- **Хранение:** один `law_articles`, bge-m3/1024, payload уже с `act`. `id` чанка уже namespace'ится по акту (`uuid5("{act}:{article}:{i}")`) — коллизий номеров между кодексами нет ✓.
- **Роутинг (агент):** `intent` возвращает вероятные акты `list[str] | None` → `search_law(query, acts)` фильтрует `MatchAny(act)`, когда уверен; иначе без фильтра (recall не теряем). Стык законов → несколько актов.
- **Быстрый путь по номеру:** явная ссылка «статья N <кодекс>» → прямой lookup по (`act`, `article_no`), минуя семантику.

**Что делаем (Роль 3):** переиндекс одного акта через **delete-by-filter `act==X`** + upsert (обновление одного закона не трогает остальные, снятые статьи чистятся). Запись `act` уже есть. Перед активацией — spike на парсинг ЗоЗПП/СК (структура HTML ИПС может отличаться от ТК). Черновик `ACTS_DRAFT` в `pipeline/acts.py` закомментирован, nd не заполнены (сверить с pravo.gov.ru), в бота НЕ заливаем.

**Важно другим:**
- **Роль 4:** payload-индекс на `act` (и `article_no`) в `init_qdrant`; опциональный параметр `act` в `search_law` (аддитивно — без него всё как сейчас).
- **Роль 2:** `intent` отдаёт предполагаемый `act`(ы) → проброс в `search_law`; не уверен → `None`. Опционально — быстрый lookup по явному номеру статьи.
