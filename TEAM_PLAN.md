## 1. Что мы строим

**Telegram-бот, который отвечает на юридические вопросы строго по законам РФ, с обязательной цитатой статьи-источника.** Пользователь может загрузить свой документ (договор, претензию) — бот учтёт его при ответе. Если данных не хватает, бот задаёт уточняющий вопрос, а не выдумывает.

Стек (черновой, менять можно, но через общее решение): **aiogram 3.x** (бот) + **LangGraph** (оркестратор агента) + **Qdrant** (векторный поиск) + **PostgreSQL** (сессии, история, версии законов) + **облачный LLM API** (только генерация ответов) + **локальные эмбеддинги** (sentence-transformers — бесплатно и не зависит от доступности внешнего API). Инструменты агента — обычные функции, **без MCP**. Всё в **docker-compose**, позже — один VPS на Selectel.

**Скоуп MVP** (этого файла): вопрос → поиск по базе законов → ответ с цитатой; загрузка PDF/фото документа; база хотя бы одного кодекса (ТК РФ). Пуши об изменении норм, мониторинг, деплой — фаза 2 (раздел 6), в MVP их НЕ тащим.

## 2. Правила параллельной работы

### Ветки и мерж

- `main` — всегда рабочий. Мерж только через PR.
- Каждый работает в своих ветках: `dev1/*`, `dev2/*`, `dev3/*`, `dev4/*` (например `dev2/langgraph-skeleton`).
- PR в `main` мержит автор после ревью любого одного из команды (или без ревью, если PR трогает только собственные папки).

### Карта владения папками

| Папка | Владелец | Что там |
|---|---|---|
| `bot/` | **Роль 1** | aiogram: handlers, FSM, форматирование |
| `agent/` | **Роль 2** | LangGraph: граф, инструменты, промпты |
| `pipeline/` | **Роль 3** | fetch/parse/chunk/embed/upsert законов |
| `infra/` | **Роль 4** | docker-compose, миграции, скрипты |
| `shared/` | **Роль 4** | контракты (Pydantic-модели, сигнатуры) |
| `tests/legal_cases/` | **Роль 2** | 15–20 эталонных юрвопросов для ручной сверки |
| корень (`CLAUDE.md`, `README.md`, `.env.example`) | **Роль 4** | правила и запуск |

Названия папок из этой таблицы — канонические: README и любые другие документы команды приводим к ним, а не наоборот.

**Главное правило: никто не редактирует чужие папки.** Нужно изменение у соседа — issue или сообщение владельцу, он делает сам. Это правило прописывается в корневом `CLAUDE.md`, чтобы его видели все четыре клауда.

**Контракты (`shared/contracts.py`) меняются только через PR с тегом всех четверых.** Пока PR не смержен, контракт считается старым.

## 3. Контракты между зонами

Зафиксированы здесь, чтобы все стартовали одновременно и мокали соседей, не дожидаясь их кода. Роль 4 переносит их в `shared/contracts.py` в день 1; до этого все пишут по этому разделу.

### 3.1 Бот ↔ Оркестратор (потребитель: Роль 1, поставщик: Роль 2)

```python
async def answer_question(user_id: int, text: str) -> Answer: ...
async def ingest_document(user_id: int, file_bytes: bytes, mime: str) -> IngestResult: ...

class Answer(BaseModel):
    text: str                          # готовый текст ответа, простым языком
    citations: list[Citation]          # минимум 1, если refused=False
    refused: bool = False              # True = честный отказ (данных нет)
    clarifying_question: str | None    # если задан — бот показывает его вместо ответа

class Citation(BaseModel):
    act: str                           # "ТК РФ"
    article: str                       # "81"
    revision_date: date
    source_url: str | None

class IngestResult(BaseModel):
    doc_id: str
    chunks: int
    ok: bool
    error: str | None
```

### 3.3 Qdrant (владелец схемы: Роль 4; пишет: Роль 3; читает: Роль 2)

| Коллекция | Payload | Примечание |
|---|---|---|
| `law_articles` | `act, article_no, chapter, status, effective_date, text` | общая база законов |
| `user_documents` | `user_id, doc_id, chunk_no, text, uploaded_at` | изоляция: фильтр по `user_id` обязателен в каждом запросе; TTL |
| `law_articles_dev` | как `law_articles` | песочница Роли 3, никто больше её не читает |

### 3.4 PostgreSQL (владелец схемы и миграций: Роль 4)

| Таблица | Поля (кратко) |
|---|---|
| `users` | `user_id PK, tg_username, consent_at (152-ФЗ, NULL = согласия нет), created_at` |
| `dialog_history` | `id PK, user_id FK, role, text, citations JSONB, created_at` |
| `law_versions` | `id PK, act, article_no, revision_date, hash, status` |
| `confidence_log` | `id PK, question, answer_id, confidence FLOAT, created_at` |

### 3.5 Формат ответа бота (владелец: Роль 1)

MarkdownV2; структура всегда: ответ простым языком → блок «Основание: ст. N …» → одна строка дисклеймера «Не является юридической консультацией». Один формат для всех пользователей.

## 4. Роли

---

### Роль 1 · Bot Layer — папка `bot/`

**Цель:** пользователь в Telegram общается с ботом, бот принимает вопросы и файлы, красиво отвечает.

**Задачи MVP:**
1. Каркас aiogram 3.x: long polling для разработки (webhook — фаза 2), `/start` (обязательное согласие на обработку ПДн, 152-ФЗ — без согласия бот не отвечает), `/help`, `/delete_my_data`.
2. FSM состояний диалога: обычный вопрос / ожидание уточнения / загрузка файла.
3. Приём текстового вопроса → вызов `answer_question()` → показ ответа.
4. Обработка `clarifying_question` (бот переспрашивает) и `refused` (честное «не нашёл» без выдумок).
5. File Handler: приём PDF/фото, валидация ДО обработки (размер ≤ 20 МБ, magic bytes), вызов `ingest_document()`, ответ «документ принят, N фрагментов».
6. Response Formatter: MarkdownV2 по формату 3.5, экранирование спецсимволов, цитаты, дисклеймер.
7. Запись диалога в `dialog_history` (через простой репозиторий поверх схемы 3.4).
8. Rate-limit на пользователя (счётчик в Postgres, N запросов/час, вежливый отказ).

**Мок на старте:** файл `bot/mock_agent.py`, реализующий `answer_question`/`ingest_document` с фикстурными ответами (1 нормальный ответ с цитатой, 1 отказ, 1 уточняющий вопрос). Заменяется на реальный `agent/` в точке И1.

**Definition of Done:** живой бот в Telegram отвечает мок-ответом с цитатой и дисклеймером, принимает PDF, переспрашивает и отказывает по фикстурам.

**Не делать:** не лезть в промпты и логику агента (`agent/`), не менять схемы БД, не трогать docker-compose (запросить Роль 4).

---

### Роль 2 · Agent Orchestrator — папка `agent/`

**Цель:** по вопросу пользователя собрать релевантные статьи, проверить цитаты и сгенерировать честный ответ.

**Задачи MVP:**
1. Граф LangGraph: `intent_classifier → retrieve → verify_citation → insufficient_context? → compose_answer`.
2. `intent_classifier`: нормализация вопроса, определение отрасли права (можно одним LLM-вызовом).
3. `retrieve`: вызов `search_law()` (3.2), отбор топ-K фрагментов.
4. `verify_citation`: каждая ссылка в черновике ответа сверяется через `verify_citation()`; несуществующие статьи — регенерация или отказ. Реализация инструментов `verify_citation` и `parse_pdf` — твоя (живут в `agent/tools/`); низкоуровневый `search_law` даёт Роль 4.
5. Ветка `insufficient_context`: если найденного мало — `clarifying_question`; если и после уточнения пусто — `refused=True`.
6. `compose_answer`: промпт с найденными статьями → LLM API → `Answer` строго по контракту 3.1. Правило промпта: отвечать ТОЛЬКО по переданным статьям.
7. `parse_pdf` + `ingest_document`: разбор файла, чанки → `user_documents` (изоляция по `user_id` — фильтр обязателен).
8. Подсчёт confidence (доля ответа, покрытая найденными фрагментами) → запись в `confidence_log`.
9. Юнит-тесты графа на фикстурах (без реального LLM — замоканный клиент).
10. `tests/legal_cases/`: 15–20 эталонных юридических вопросов с ожидаемыми статьями-ответами — ручная сверка качества перед демо (единственный способ доказать точность на защите).

**Мок на старте:** свой скрипт `agent/fixtures/load_fixtures.py` — 20–30 статей ТК/ГК руками в коллекцию `law_articles` локального Qdrant (поднятого через docker-compose Роли 4; до его готовности — Qdrant из одиночного `docker run`, команда в README фикстур).

**Definition of Done:** `answer_question("могут ли уволить в отпуске?")` возвращает `Answer` с реальной цитатой ст. 81 ТК РФ из фикстур; на бессмысленный вопрос — уточнение или отказ.

**Не делать:** не парсить настоящие источники НПА (это Роль 3), не писать хендлеры бота, не менять контракты в одиночку.

### Роль 3 · Data Pipeline — папка `pipeline/`

**Цель:** настоящие тексты законов автоматически попадают в Qdrant с метаданными редакций.

**Задачи MVP:**
1. **Spike (день 1, таймбокс):** выбрать источник НПА — сравнить официальные порталы (pravo.gov.ru и др.) по: полнота, формат (HTML/PDF), стабильность разметки, условия использования. Результат — `pipeline/SOURCE.md` с решением. Это открытый вопрос всего проекта — решаешь ты.
2. `fetch`: скачивание текста акта из выбранного источника.
3. `parse`: TXT/HTML → структура «глава → статья → пункт»; PDF → текст (+ OCR для сканов, tesseract).
4. `chunk`: разрез строго по статьям/пунктам, метаданные: `act, article_no, chapter, status, effective_date`.
5. `embed`: эмбеддинги локально через sentence-transformers (например, `intfloat/multilingual-e5-base`; имя модели — из `.env`), батчами; CPU достаточно, GPU не нужен.
6. `upsert` в **`law_articles_dev`** (свою песочницу; переключение на боевую `law_articles` — точка И2, по команде в общем чате).
7. Версионирование: `hash` текста статьи → `law_versions` (Postgres); повторный прогон обновляет только изменившееся.
8. CLI-запуск: `python -m pipeline.run --act tk_rf` (cron/Airflow — фаза 2).

**Мок на старте:** ничего мокать не нужно — пайплайн самодостаточен. Qdrant/Postgres — локальный docker.

**Definition of Done:** ТК РФ целиком лежит в `law_articles_dev` с корректными метаданными; повторный запуск не плодит дубликатов; `SOURCE.md` написан.

**Не делать:** не трогать логику поиска (`search_law` — Роль 2/4), не писать в боевую `law_articles` до точки И2, не менять схему коллекций (PR к Роли 4).

### Роль 4 · Инфраструктура и хранилища — папки `infra/`, `shared/`, корень

**Цель:** у любого из четверых всё поднимается одной командой; контракты и схемы живут в коде.

**Задачи MVP (первые две — в день 1, они разблокируют остальных):**
1. **Скелет репозитория**: папки из раздела 2, `pyproject.toml` (общие зависимости: pydantic, aiogram, langgraph, qdrant-client, asyncpg, sentence-transformers, pytest), `.gitignore`, `README.md` (как запустить).
2. **`shared/contracts.py`**: перенос раздела 3 в код, 1-в-1. После мержа — контракт живёт в коде, этот файл становится историей.
3. **Корневой `CLAUDE.md`**: карта владения папками из раздела 2, правило «не редактируй чужие папки», «не меняй shared/ без PR», ссылка на TEAM_PLAN.md.
4. `docker-compose.yml`: qdrant (+volume), postgres:16 (+volume), bot-сервис (build из репо); порты БД не наружу.
5. Миграции Postgres (alembic или простые .sql в `infra/migrations/`) по схеме 3.4.
6. Скрипт `infra/init_qdrant.py`: создание коллекций по схеме 3.3 (включая `_dev`).
7. `.env.example`: `BOT_TOKEN, LLM_API_KEY, LLM_MODEL` (например, `claude-sonnet-5`), `EMBED_MODEL` (имя sentence-transformers-модели), `POSTGRES_*`, `HTTPS_PROXY` (опционально, см. задачу 11) — с комментариями.
8. `Makefile`: `make up`, `make migrate`, `make test`, `make fixtures`.
9. Реализация `search_law()` нижнего уровня (запрос в Qdrant с фильтрами по схеме 3.3) — как библиотечная функция в `shared/`, которую Роль 2 зовёт из графа. (Если спорно — договориться с Ролью 2, кто пишет; по умолчанию — Роль 4, т.к. это работа со схемой хранилища.)
10. GitHub: создать репозиторий, ветки, включить обязательные PR в `main`.
11. **Проверить доступность LLM API с ваших IP (день 1):** Anthropic/OpenAI из РФ напрямую могут быть недоступны — если блок, настроить прокси (`HTTPS_PROXY` в `.env`) и задокументировать в README. Это влияет на всех — сообщить результат в чат.

**Definition of Done:** свежий клон + `.env` + `make up && make migrate` → всё работает у каждого из четверых; `CLAUDE.md` лежит в корне.

**Не делать:** не писать промпты, хендлеры и парсеры — только каркас, схемы, склейка.

## 5. Интеграция: кто кого ждёт

**Никто никого — до точек интеграции.** До них все работают на моках/фикстурах из своих разделов.

| Точка | Что происходит | Кто участвует | Когда |
|---|---|---|---|
| **И0** | Скелет + контракты в `main` | Роль 4 | конец дня 1 |
| **И1** | `bot/mock_agent.py` → реальный `agent/` | Роли 1+2 | оба DoD готовы |
| **И2** | `law_articles_dev` → боевая `law_articles`; фикстуры Роли 2 больше не нужны | Роли 2+3 | DoD Роли 3 готов |
| **И3** | Всё вместе через `docker compose up`, сквозной тест: вопрос в Telegram → ответ с цитатой из настоящего ТК РФ | все | финал MVP |

**Синхронизация:** одно сообщение в общий чат в конце дня — «что смержил в main, что блокирует». Всё остальное — асинхронно через PR.

**Если клауды упёрлись друг в друга** (два PR меняют один файл) — значит, нарушена карта владения из раздела 2; файл отдаётся владельцу, второй PR переделывается.
